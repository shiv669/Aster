# Aster

**Aster** is an AI powered universal CSV ingestion engine that converts arbitrary spreadsheets into validated CRM records.

The name is intentionally short, memorable, and product like. It does not reveal implementation details, allowing the engine to evolve into imports from Excel, Google Sheets, Notion exports, CRMs, APIs, and databases without renaming.

---

# Product Vision

Most CSV importers assume the user already knows the schema.

Aster assumes the opposite.

Users upload anything.

The system figures everything else out.

The intelligence is not in reading CSVs.

The intelligence is understanding messy business data.

Every upload follows a deterministic pipeline where AI is used only where reasoning is required.

Everything else is traditional software engineering.

---

# Core Philosophy

The system should behave as though it were replacing an employee whose only job is importing CRM leads all day.

It should never blindly trust AI.

It should never blindly trust CSVs.

Every stage should validate the previous stage.

Every output should be explainable.

Every failure should be recoverable.

---

# Engineering Principles

A solution that another engineer understands in two minutes is better than one requiring twenty minutes.

---

## AI is one component

If software can solve something without AI,

Every file should be understandable in under five minutes.

Every function under roughly thirty lines unless complexity genuinely requires more.

Every function should have one job.

Never nest deeply.

Return early.


# Code Style Guide

## Rule 1

Write code that is obvious.

Not clever.

Bad

```ts
const r=a?.b?.c?.map(x=>f(x)).filter(Boolean)
```

Good

```ts
const users = response.users;

const activeUsers = users
    .map(transformUser)
    .filter(Boolean);
```

Future maintainers matter more than saving three lines.

---

## Rule 2

Prefer composition.

Never inheritance.

Bad

```text
AIImporter

↓

CRMImporter

↓

GoogleImporter

↓

FacebookImporter
```

Good

```text
Importer

↓

Parser

↓

Validator

↓

Normalizer

↓

Exporter
```

Small reusable building blocks.

---

## Rule 3

Every function should answer one question.

Bad

```text

Uploads

Parses

Repairs

Calls AI

Downloads JSON
```

Good

```text

↓

↓

detectSchema()

↓

normalize()

↓

runAI()

↓

validate()

↓

export()
```

---

## Rule 4

Pure functions wherever possible.

Good

```ts

```

Bad

```ts
normalizePhone(phone, database, logger, cache)
```

Pure functions are easier to test.

# The One Rule That Governs Everything

Every line of code should earn its place.

If removing a line changes nothing, delete it.

If a dependency saves only three lines, don't install it.

If an abstraction is needed only once, don't create it.

If an AI call can be replaced with deterministic logic, remove the AI call.

The final codebase should feel surprisingly small for what it accomplishes. That is usually a sign of a mature engineering design.

# The Aster Standard

Every feature added to Aster must satisfy five questions before it is accepted.

**Useful.**

Does it solve a real problem?

**Simple.**

Can a new engineer understand it quickly?

**Reliable.**

Does it fail safely?

**Replaceable.**

Can its implementation be swapped without changing the rest of the system?

**Explainable.**

Can both the user and the developer understand why it behaved the way it did?

If a feature cannot satisfy all five, it does not belong in Aster.

This standard becomes the filter that keeps the codebase small, maintainable, and capable of growing for years without collapsing under its own complexity.
